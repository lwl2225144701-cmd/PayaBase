# 自治 Agent 实现优先级清单

## 目的

把工程实现拆成可执行顺序，避免一次性重构过大，也避免先做末端功能、后补基础状态。

## 第一优先级：先把执行骨架搭起来

必须先做：

- `AgentOrchestrator`
- `AgentPlanner`
- `AgentRunner`
- `AgentStepExecutor`
- `AgentState`
- `AgentPolicy`

原因：

- 没有总控层，自治 Agent 只是“更复杂的聊天”。
- 没有状态层，模型容易忘记当前步骤。
- 没有 policy，步骤会失控。

验收标准：

- 能创建一个 run
- 能生成一个初始计划
- 能推进到第一步执行
- 能把第一步结果写回状态

## 第二优先级：先把记忆做稳

必须先做：

- `plan_snapshot`
- `current_step`
- `next_step`
- `completed_steps_summary`
- `step_history`
- 步骤结果压缩逻辑

原因：

- 你指出的核心风险就是“做着做着忘了下一步”。
- 这个问题不先解决，后面接 PPT / PDF 只会更乱。

验收标准：

- 多步任务执行时，模型每轮都知道自己在做哪一步
- 已完成内容不会在下一轮丢失

## 第三优先级：先把方案型任务跑通

必须先做：

- 知识检索
- 方案生成
- 方案型任务的自治闭环

建议顺序：

1. 先查资料
2. 再生成方案
3. 再返回结构化结果

原因：

- 这是最小闭环
- 也是当前最稳定、最容易验收的场景

验收标准：

- 一条复杂方案请求可以自动完成
- 结果格式稳定
- 失败可解释

## 第四优先级：再接产物任务

必须后做：

- `ppt_generation`
- `pdf_export`

原因：

- 产物任务是异步链路，涉及状态、任务 ID、前端轮询。
- 在主链路没稳之前，先接产物容易把调试复杂度拉高。

验收标准：

- 同一 Agent 主链路可以提交 PPT / PDF
- `task_id` 能正常返回
- 异步失败不会卡死主流程

## 第五优先级：再扩到通用场景

最后再做：

- 知识问答
- 文档总结
- 内容生成
- 现有 `RequestRouter` 规则内部化

原因：

- 先把复杂任务链路跑稳，再把更多请求收进来
- 否则很容易把自治 Agent 做成“全量大路由器”

验收标准：

- 全量能力都能进同一 Agent
- 对外只保留一条主链路

## 不建议提前做的事

- 不要先做完整前端大面板
- 不要先做所有工具的重构
- 不要先把所有路由规则都挪进自治层
- 不要先追求无限自治和完全开放式规划

## 推荐开发节奏

1. 先做骨架
2. 再做记忆
3. 再做方案闭环
4. 再接产物
5. 最后扩场景

## 最终判断标准

如果某一步不能回答这三个问题，就不要进入下一步：

- 这一步的输入是什么？
- 这一步的输出是什么？
- 这一步失败后怎么处理？
