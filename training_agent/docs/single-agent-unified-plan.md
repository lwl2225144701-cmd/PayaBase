# 单一自治 Agent 统一方案

## Summary
保留你们已经测通的完整请求能力，但把这些能力全部收进一个新的自治 Agent 里，对外只保留一条主链路：`用户输入 -> 新 Agent -> 结果 -> 前端`。  
现有经过测试的路由规则、RAG、方案生成、PPT、PDF、内容生成能力不再作为第二套体系存在，而是变成这个新 Agent 内部的能力模块。

## Key Changes
- 新建唯一的自治 Agent 入口：
  - 所有请求先进入新 Agent。
  - 不再对外保留“现有路由链路”和“自治链路”两套并行入口。
- 现有完整请求流整体内化：
  - 你们已经测试通过的请求理解、路由、RAG、生成、产物输出逻辑，全部作为 Agent 内部的能力复用。
  - `RequestRouter` 不再是独立业务分流器，而是 Agent 内部的决策组件。
- `ReActAgent` 继续复用，但角色收窄：
  - 只负责 Agent 内部某一步的多工具执行。
  - 不承担对外主入口，也不承担全局编排。
- 新增自治总控层：
  - `AgentOrchestrator` 管整个运行周期。
  - `AgentPlanner` 负责拆解目标。
  - `AgentRunner` 负责逐步执行。
  - `AgentState` 负责保存 run/step 状态、产物和错误。
- 统一输出：
  - 最终答案
  - 中间步骤
  - 产物任务 ID
  - 失败与回退信息
  - 进度状态

## Implementation Changes
- 顶层只保留一个 Agent 流程：
  - `chat` 入口 -> `AgentOrchestrator`
- 现有请求路由能力下沉为内部模块：
  - 作为 Agent 的“意图理解 + 动作选择”能力使用
  - 不再对外单独分流到旧链路
- 现有工具不重写：
  - RAG、方案、PPT、PDF、内容生成继续作为 Agent 的工具
  - `ToolRegistry` 保持统一接入
- 旧业务链路收敛：
  - 不再维护两套并行的“现有流”和“自治流”
  - 现有验证通过的逻辑保留，但只作为新 Agent 的内部实现
- 状态与护栏：
  - 单 run 最大步数
  - 总超时
  - 重试上限
  - 工具白名单
  - 失败回退策略

## Test Plan
- 主链路测试：
  - 所有请求都从新 Agent 入口进入
  - 验证不再存在对外两套并行链路
- 能力回归测试：
  - 验证原来已测通的 RAG、总结、方案、PPT、PDF、内容生成，都能在新 Agent 内正常工作
- 执行链路测试：
  - 验证 Planner -> Runner -> ReActAgent 的多步执行闭环
- 护栏测试：
  - 验证步数、超时、重试、工具白名单都能生效
- 前端测试：
  - 验证前端能显示自治过程、步骤和产物，不只是最终文本

## Assumptions
- 对外只保留一条 Agent 主链路，不再维护两套业务体系。
- 现有经过测试的请求流保留为内部能力实现，而不是独立入口。
- `RequestRouter` 逻辑保留，但只作为 Agent 内部能力复用。
- `ReActAgent` 保留，但只作为步骤级执行器。
